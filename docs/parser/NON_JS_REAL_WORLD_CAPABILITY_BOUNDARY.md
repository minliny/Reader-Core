# Non-JS Real World Capability Boundary

## 1. 已验证支持

### 核心能力
- **非 JS 解析**：支持纯静态 HTML 解析，无需 JavaScript 执行
- **无登录依赖**：所有测试用例均不要求登录
- **完整 pipeline**：支持 search → detail → toc → content 全链路解析
- **静态 HTML**：支持标准 HTML 结构的解析

### 选择器能力
- **简单类选择器** (`css:.class`)：支持通过 class 属性选择元素
- **ID 选择器** (`css:#id`)：支持通过 ID 属性选择元素
- **标签选择器** (`css:tag`)：支持通过 HTML 标签选择元素
- **子选择器** (`css:.parent > .child`)：支持直接子元素选择
- **标签+类选择器** (`css:tag.class`)：支持标签与类组合选择

### 解析特性
- **文本提取**：支持从选定元素中提取文本内容
- **链接提取**：支持从 `<a>` 标签中提取 href 属性
- **图片提取**：支持从 `<img>` 标签中提取 src 属性
- **内容清理**：支持基本的空白字符清理
- **结构一致性**：支持确保解析结果的结构一致性

## 2. 明确不支持

### 技术限制
- **JS 渲染**：不支持需要 JavaScript 执行才能生成内容的页面
- **登录/鉴权**：不支持需要登录才能访问的内容
- **动态签名**：不支持需要动态生成签名的 API 接口
- **加密接口**：不支持加密的内容接口
- **高级 CSS 选择器**：不支持属性选择器、伪类选择器等高级 CSS 选择器
- **动态键**：不支持动态生成的字段名
- **泛化能力**：不支持 ≥3 级的泛化处理，除非已有明确实现

### 功能限制
- **网络请求**：仅使用本地 fixture 进行测试，不执行实际网络请求
- **复杂规则**：不支持复杂的规则格式和转换逻辑
- **多页内容**：不支持自动处理分页加载的内容
- **动态内容**：不支持需要滚动或点击才能加载的内容

## 3. 支持边界判定规则

### 可进入 parser backlog 的失败
- **选择器语法错误**：标准 CSS 选择器语法错误
- **解析逻辑缺陷**：解析过程中的逻辑错误
- **结构处理问题**：HTML 结构处理不当
- **性能问题**：解析性能异常

### 应标记为 unsupported 的失败
- **JS 依赖**：页面明确依赖 JavaScript 渲染
- **登录要求**：内容明确需要登录
- **加密内容**：内容经过加密处理
- **动态接口**：API 接口需要动态签名
- **高级选择器**：使用了不支持的高级 CSS 选择器

### Fixture / Expected 问题
- **Fixture 格式错误**：HTML fixture 格式不正确
- **Expected 数据错误**：预期结果数据与实际解析逻辑不匹配
- **路径问题**：文件路径或结构不正确
- **编码问题**：字符编码不一致

## 4. 与 regression_matrix 的关系

### 矩阵作用
- **能力覆盖**：regression_matrix 记录每个测试用例使用的 parser 特性
- **状态跟踪**：跟踪每个测试用例的执行状态
- **失败分类**：根据 failure taxonomy 对失败进行分类
- **趋势分析**：通过历史数据分析 parser 能力演进

### 矩阵维护
- **新增能力**：当 parser 新增能力时，需更新矩阵中的 `parser_features_used` 字段
- **失败分析**：当测试失败时，需更新 `failure_reason` 和 `failure_taxonomy` 字段
- **状态更新**：定期更新测试用例的 `status` 和 `stage_results` 字段
- **验证命令**：记录用于验证测试用例的命令

### 能力边界与矩阵的对应关系
- **支持能力**：在矩阵中通过 `parser_features_used` 体现
- **不支持能力**：在矩阵中通过 `failure_taxonomy` 体现为 `unsupported` 类型
- **边界情况**：通过 `known_gaps` 字段记录已知的能力边界
- **验证状态**：通过 `verification_status` 字段记录验证状态

## 5. 能力边界扩展流程

1. **需求分析**：分析新的解析需求
2. **能力评估**：评估当前 parser 是否支持
3. **测试用例**：创建针对新能力的测试用例
4. **实现与验证**：实现新能力并验证
5. **矩阵更新**：更新 regression_matrix 记录新能力
6. **文档更新**：更新本能力边界文档

## 6. 结论

当前 Reader-Core 的非 JS 解析能力已达到工程化基线要求，能够稳定解析标准的静态 HTML 书源。通过严格的 regression 测试和能力边界定义，确保了解析能力的可预测性和稳定性。

未来的能力扩展将基于实际需求和测试结果，遵循上述边界判定规则，确保解析能力的可靠性和可维护性。
